# Picker Row Styling Plan

## Goal
Normalize picker entry rendering so every row uses one consistent spacing pattern and a simpler, more uniform style. The intended result is that navigation highlight looks consistent instead of showing patchy spacing/background artifacts across row segments.

## Findings
- Picker labels are built in `plugin/ws/ui.lua` and consumed by `plugin/ws/selectors.lua`.
- The current inconsistency comes from manual spacing in multiple formatters:
  - `append_icon_and_text()` always appends a trailing space
  - `format_live_workspace_label()` adds extra literal spaces around pane counts and the current marker
  - `format_directory_label()` bakes a trailing space into the basename segment
  - `format_saved_workspace_label()` uses `string_format(' %s ', saved_name)`
- `plugin/ws/selectors.lua` only assembles choices and routes actions; it is not the source of the row styling problem.
- The repo does not currently set picker row backgrounds. The white selected-row background is likely WezTerm `InputSelector` behavior, so the plugin-side fix should focus on producing uniformly formatted labels that highlight cleanly.

## Recommended Implementation
1. Refactor row assembly in `plugin/ws/ui.lua` around one shared spacing rule.
   - Replace the current trailing-space helper with a small helper that appends logical row parts without embedded leading/trailing spaces.
   - Insert separators between visible parts centrally so every formatter uses the same spacing model.
2. Update the label formatters in `plugin/ws/ui.lua` to use the same layout shape.
   - `format_action_label()`
   - `format_live_workspace_label()`
   - `format_directory_label()`
   - `format_saved_workspace_label()`
   - Target layout: `[prefix] [primary text] [secondary metadata] [status]`
3. Simplify row styling so each row reads as one coherent entry.
   - Keep accent colors only for genuinely separate parts such as the prefix, metadata, and current marker.
   - Keep primary text on `config.colors.text`.
   - Avoid extra inline color switches inside a single logical phrase.
4. Remove the current spacing quirks rather than preserving them.
   - No leading padding around saved workspace names
   - No formatter-specific trailing spaces
   - No doubled spaces when icons or labels are empty
   - No uneven left/right padding between row types
5. Leave selector behavior in `plugin/ws/selectors.lua` unchanged unless a formatter signature change forces a trivial call-site adjustment.

## Critical Files
- `plugin/ws/ui.lua`
- `plugin/ws/selectors.lua`
- `docs/configuration.md`
- `README.md`
- `examples/custom.lua` (only if a small wording/comment update is useful)

## Verification
1. Open the main workspace selector in WezTerm and inspect:
   - action rows
   - live workspace rows
   - zoxide rows
   - saved workspace rows in restore/delete pickers
2. Confirm every row uses the same spacing rhythm:
   - no leading padding
   - no trailing padding
   - exactly one separator space between visible parts
   - no doubled spaces when optional icons or labels are blank
3. Confirm navigation no longer produces patchy segment-by-segment row styling.
   - Any remaining full-row white highlight should be treated as WezTerm `InputSelector` behavior, not plugin background logic.
4. Check live workspace details:
   - pane count metadata stays compact and consistently spaced
   - current workspace marker renders as one clean trailing part
5. Check saved workspace details:
   - saved names are no longer wrapped in manual spaces
   - optional timestamp / cwd metadata appears as one compact secondary block
6. Exercise selector actions to ensure labels still map to the correct choice ids and behavior:
   - switch live workspace
   - create from zoxide entry
   - restore saved workspace
   - delete saved/live workspace
7. Update docs text only where it describes selector colors/layout so it matches the new uniform row rendering.
